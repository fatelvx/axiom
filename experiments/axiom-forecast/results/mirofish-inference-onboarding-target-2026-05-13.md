3. Strongest adoption improvement.
The `no_spec_files` dead-end fix. It is the difference between a user feeling "tool broken" and "tool is guiding me to a concrete next action." For a cold start, this is the only part of the change that directly reduces immediate abandonment.

4. Remaining rejection pattern.
The output of `axi infer` still looks like a dependency graph tool that prints comments. Users accustomed to architecture advice (even lightweight) will note that it surfaces structure but does not surface *intent*. The absence of any language about layering direction, stability, or typical patterns (even in advisory form) leaves them asking "so what?" Skeptical evaluators will dismiss the tool as yet another cycle-detector with a contract file format. The "review" prompt without architectural guidance may heighten anxiety for novices and feel like hand-holding for experts.

5. Decision.
Run one more small refinement before a larger backtest. The current change is a necessary patch, but user comprehension of the starter contract as an architecture aid is still too fragile. A single targeted improvement to the `axi infer` output can raise confidence that a wider backtest will produce useful qualitative feedback instead of "it found cycles, I guess."

6. Recommended next step.
Add a short, non-enforced, human-readable summary comment at the top of the generated `.axi` file that explains what a starter contract is and is not. For example:
```
# starter contract generated by `axi infer`.
# This is an inference of the current dependency graph, not a recommended architecture.
# Review the collapsed cycles: they may indicate intentional grouping or a design issue.
# Once you have decided on module boundaries, rename, split, or merge as appropriate,
# then use `axi check` to enforce your intended contracts.
```
Also, ensure the JSON output includes a `starterContract` key with a `notice` field containing this same text, so AI agents and CI tooling can present it. This single change addresses the "why does this matter" gap without adding enforcement semantics. It costs a few developer-hours and can be verified in a quick smoke test. After this, a larger backtest with pilot users becomes reasonable.

7. What not to overclaim.
Do not claim that this change makes `axi infer` feel like "architecture observability" to skeptics. It remains an improved dependency analyzer output. Do not claim that `ServicesCycle` naming resolves confusion about strongly connected components; it only reduces surface-level readability friction. Do not claim that the `no_spec_files` hint solves onboarding entirely; some users will still need to understand `.axi` contracts before they can act on the suggestion. Do not claim that AI agents now get "architecture feedback"; they get structured dependency data with human readable notes, which is useful but not semantic architectural reasoning.
