# Evidence Artifact Loop

Axiom should give local developers, PR reviewers, CI, and agents the same architecture evidence.

The boring first artifact is not a new file format. It is a convention around files and commands that already exist:

```text
.axi contract + scan scope + graph baseline + review story + visible intentional debt
```

This is the single-repository trust loop that future GitHub Actions, VS Code, MCP, contract templates, and inheritance workflows should wrap. If an integration needs different validation semantics, it is probably too early.

## Artifact Parts

Use these parts together:

| Part | Typical location | Purpose |
| --- | --- | --- |
| `.axi` contract | `axiom/main.axi` or `contracts/<project>.axi` | Declared architecture intent, module ownership, allowed edges, visibility rules, and visible intentional debt. |
| Scan scope | `axiom.config.json` or CLI `--include` / `--exclude` | Keeps source discovery aligned with the architecture question. |
| Graph baseline | `.axi/baselines/current.graph.json` | Unfiltered `axi graph --json` snapshot used to review observed-edge drift. |
| Review story | generated by `axi observe --json` or `--markdown` | Short explanation of hard failures, visible debt, warnings, drift, and what to inspect first. |
| Intentional debt ledger | `.axi` `accepts ... until ... because ...` plus JSON `intentionalDebt[]` | Temporary architecture debt that stays visible and expires. |
| Starter inference evidence | generated draft comments or `axi infer --json` | Temporary authoring evidence while reviewing an inferred contract, especially collapsed cycles, cycle-breaking candidates, dependency evidence, and large-file pressure notes. |

The contract is the source of truth. The baseline is a snapshot. The review story is generated evidence. Intentional debt belongs in `.axi`, not in hidden allowlists, PR comments, or agent memory.

Starter inference evidence is not a second contract. It matters only while turning an inferred draft into reviewed intent.

## Recommended Layout

For an adopted repository:

```text
axiom/
  main.axi
.axi/
  baselines/
    current.graph.json
axiom.config.json
```

For an external pilot, keep the contract outside the target repository:

```text
contracts/
  target-app.axi
target-artifacts/
  current.graph.json
```

When using an external contract, pass it with `--spec`. Paths inside the contract still point at files under `--root`.

## First Adoption Loop

Start from evidence, then decide what should become intent:

```bash
axi infer --root . > axiom-starter.axi
axi observe --root . --spec axiom-starter.axi --markdown
axi graph --root . --spec axiom-starter.axi --json > .axi/baselines/current.graph.json
axi diff .axi/baselines/current.graph.json --root . --spec axiom-starter.axi
```

The inferred contract mirrors the current graph. Review module names, dependency evidence, collapsed cycles, visibility suggestions, and architecture pressure notes before treating it as declared intent.

If `axi infer` collapses a cycle, keep the collapsed-cycle evidence in the review. The generated contract intentionally merges those source groups so `axi graph` and `axi diff` will not show edges inside the merged module later. The cycle path samples, internal edges, and cycle-breaking candidates are the evidence you need before deciding whether to keep the merged module, split it, or make an edge explicit.

After review, move only the contract you understand into `axiom/main.axi`. Keep the baseline only after the team agrees the observed graph shape is intentional enough to watch.

## Local Development Loop

Use `axi check` for facts that are explicit enough to gate:

```bash
axi check --root .
```

Use `axi observe` for review context:

```bash
axi observe --root . --baseline .axi/baselines/current.graph.json --markdown
```

If there is no baseline yet, run `axi observe --root . --markdown` and say that drift review is not available. Do not create a fresh baseline during the same change you are reviewing.

## PR Loop

A PR integration should keep the split visible:

```text
axi check --json = hard gate
axi observe --json = review story, warnings, visible debt, drift
```

The PR should fail only on hard `violations[]` unless the team has explicitly added another policy. Advisory warnings, graph interpretation, and baseline drift should be review context first.

Good PR summary shape:

```text
Gate: pass or fail from axi check
Story: architectureSummary.reviewStory.summary
Hard violations: summary.violations
Visible debt: summary.intentionalViolations
Warnings: summary.warnings
Drift: drift.newObservedEdges + drift.removedObservedEdges
Next step: architectureSummary.reviewStory.nextStep
```

## Agent And MCP Loop

Agents should read the same artifact instead of relying on remembered architecture rules:

1. Run `axi check --json` after edits.
2. Repair hard `violations[]`.
3. Run `axi observe --json` for `architectureSummary.reviewStory`, `warnings[]`, `intentionalDebt[]`, and optional `drift`.
4. Propose `.axi` contract changes only when the architecture intent changed.
5. Ask for human approval before adding `accepts ... until ... because ...`.

A safe MCP v0 should be read-only over these commands. It should not auto-edit contracts, update baselines, or accept debt.

## Updating The Artifact

Update the contract when intent changes:

- a module is renamed or split,
- a dependency becomes intended architecture,
- a public entry point or hidden path is formalized,
- a temporary violation needs visible debt with an expiration date,
- an old accepted violation is removed.

Update the baseline when the observed graph shape is intentionally different:

```bash
axi graph --root . --json > .axi/baselines/current.graph.json
```

Review baseline updates like architecture changes. A baseline generated from the current PR and then compared against the same PR hides drift.

## What Not To Do

Avoid these shortcuts:

- Do not treat `axi infer` output as a contract without review.
- Do not make advisory warnings or drift fail CI by default.
- Do not hide accepted debt outside `.axi`.
- Do not let an agent add `accepts` rules without approval.
- Do not parse Markdown when JSON is available.
- Do not create a fresh baseline during PR review.
- Do not make GitHub Actions, VS Code, or MCP invent their own validation rules.

The artifact loop is intentionally plain. Axiom becomes more useful when the same evidence can travel across local runs, PRs, IDEs, and agents without changing what the validator means.
